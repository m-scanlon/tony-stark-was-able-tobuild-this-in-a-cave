# voice_event Schema

The `voice_event` is the only event type the Voice Shard sends to the Brain Shard. It carries the user's transcript, intent hints, and session state.

The envelope is assembled by the shard's transport layer before sending. The intent model only produces `triage_hints` ‚Äî all other fields are hydrated by the shard: `schema` and `ts` are stamped at send time, `turn_id` is generated by the transport layer, `transcript` comes from STT, and `session_state` comes from the shard's local turn tracking.

---

## Envelope Hydration

The intent model only produces `triage_hints`. The rest of the envelope is assembled by the shard's transport layer before the event is sent.

| Field | Source |
|---|---|
| `schema` | Transport layer ‚Äî stamped at send time |
| `turn_id` | Transport layer ‚Äî generated per turn |
| `ts` | Transport layer ‚Äî stamped at send time |
| `device_id` | Transport layer ‚Äî shard identity, stamped at send time |
| `transcript` | STT model |
| `triage_hints` | Intent model ‚Äî the only field the model produces |
| `session_state` | Shard local turn tracking |

## Fields

### `schema`
Version identifier. Always `voice_event_v1` for v1. Stamped by the shard's transport layer ‚Äî not produced by the intent model.

### `turn_id`
Unique ID for this turn. Stamped by the shard's transport layer before sending ‚Äî not produced by the intent model. Used for deduplication and outbox tracking. The brain generates its own `event_id` on ingress ‚Äî `turn_id` is the Voice Shard's reference, not the Brain Shard's.

### `ts`
ISO 8601 timestamp. Stamped by the shard's transport layer before sending ‚Äî not produced by the intent model.

### `device_id`
The identity of the shard that captured this event. Stamped by the shard's transport layer ‚Äî not produced by the intent model. Allows the brain to know which shard to route responses back to.

### `transcript`
What the user said, as plain text. Output of the Voice Shard's STT model.

### `triage_hints`
Intent classification produced by the Voice Shard's intent model. Tells the brain how to prioritize and how to instruct the UX model to respond. Each classification carries its own confidence score so the brain can independently decide how much to trust each one.

> **Note:** The fields inside `triage_hints` are not fully locked. The UX model design will determine what hints it actually needs. The examples below are directional ‚Äî treat them as placeholders until the UX layer is designed.

#### `intent`
What the user wants.

- `summary` ‚Äî one sentence describing the user's intent
- `confidence` ‚Äî float 0.0‚Äì1.0. How confident the model is in the intent classification. Low confidence may cause the brain to re-derive intent from the transcript directly rather than trusting this field.

#### `latency_class` (? ‚Äî not locked, depends on UX model design)
How urgently the user expects a response.

- `value`: `interactive | background | deferred`
  - `interactive` ‚Äî user is waiting, respond as fast as possible
  - `background` ‚Äî can run while the user does something else
  - `deferred` ‚Äî no urgency, schedule it for later
- `confidence` ‚Äî float 0.0‚Äì1.0

#### `ack_policy` (? ‚Äî not locked, depends on UX model design)
How the UX model should acknowledge the request while the brain is working. The exact values here will be defined once the UX model and its acknowledgement behavior are designed.

- `value`: `spoken_if_slow | earcon_only | silent`
  - `spoken_if_slow` ‚Äî speak a wait phrase if the response is taking time
  - `earcon_only` ‚Äî play a sound only, no words
  - `silent` ‚Äî no acknowledgement
- `confidence` ‚Äî float 0.0‚Äì1.0

### `session_state`
Responsible for managing context about outgoing jobs and syncing that context with the brain. The shard tracks what jobs are in flight on its side ‚Äî this field is how it communicates that state so the brain can route and continue correctly.

Shape is settled for v1 (`pending_job_id`, `waiting_for`). Routing semantics ‚Äî how the Brain Shard acts on this to distinguish new jobs from continuations ‚Äî are still being designed. See `next-steps.md` ¬ß2.

#### `pending_job_id` (? ‚Äî may be restructured into a job object)
- `null` ‚Äî new job, the brain should start fresh
- set to a job ID ‚Äî this turn continues an existing open job (waiting on user approval, clarification, etc.)

#### `waiting_for` (? ‚Äî may be restructured into a job object)
What the existing job is waiting on, if anything. Examples: `user_approval`, `clarification`.
