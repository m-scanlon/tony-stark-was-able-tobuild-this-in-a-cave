# Event Ingress and ACK

## Overview

The Event Ingress and ACK system guarantees reliable delivery of shard commands to the API Gateway with durable persistence and duplicate-safe handling.

**The command is the event.** No context. No envelope. No JSON payload. The only state is the command itself — `octos <tool> [args]`. The API Gateway assembles everything else from Redis after receiving it.

Reliability goals:

- no lost commands across crash/restart/network failure
- at-least-once delivery from shard to API Gateway
- idempotent duplicate handling
- ACK only after durable inbox write commit

---

## Terminology

- **Command**: `octos <tool> [args]` — the only thing a shard sends. No context attached.
- **Command Register**: durable store on the sending shard tracking commands awaiting ACK.
- **Inbox**: durable store on the API Gateway for received commands.
- **ACK**: `octos ack --turn=<turn_id> --status=stored` — sent back as a command.
- **Idempotency**: duplicate commands produce no duplicate side effects.
- **At-least-once delivery**: sender retries until ACK; duplicates handled by inbox.

---

## Architecture

```
+---------------------+       +-----------------------+       +----------------------+
| Shard               |       | Transport             |       | API Gateway          |
| command emitter     |-----> | octos <tool> [args]   |-----> | Ingress + Inbox      |
+----------+----------+       +-----------+-----------+       +----------+-----------+
           |                                  ^                           |
           v                                  |                           v
  +------------------+             octos ack --turn=...            +-------------+
  | Command Register | <------------------------------------------ | SQLite Inbox|
  +------------------+                                             +-------------+
           |
           v
  pop by turn_id only after ACK
```

---

## Responsibility Split

**Brain Shard** owns session and turn identity. It generates `session_id` and `turn_id` on receipt. Ingress shards never produce or track these identifiers.

**Ingress shards** are responsible for one thing: issuing commands. They stream raw data to the brain. The brain handles identity, session tracking, and all downstream resolution.

## Command Syntax

Shards emit only their registered primitives. A Voice Shard streams tokens and vectors — no session, no turn, no context:

```
octos stream --token="what" --valence=-0.4 --arousal=0.7 --dominance=0.5
octos stream --token="did" --valence=-0.3 --arousal=0.6 --dominance=0.5
octos stream --token="nginx" --valence=-0.5 --arousal=0.8 --dominance=0.4
```

That is the entire payload. The brain receives the stream, assigns identity, resolves everything else.

---

## ACK Protocol

ACK is a command — same syntax, other direction:

```
octos ack --turn=turn_8f4c --status=stored
```

Rules:

- ACK sent only after inbox write commits
- Sending shard pops command register entry on matching `turn_id`
- Duplicate command (same `turn_id`) → re-ACK, no reinsert
- Invalid command → NACK, remains in register for retry

---

## Inbox Schema

The API Gateway's inbox stores the raw command string and enough metadata for deduplication and retry:

```sql
CREATE TABLE IF NOT EXISTS command_inbox (
  command_id      TEXT PRIMARY KEY,      -- API Gateway-generated ULID
  session_id      TEXT NOT NULL,
  turn_id         TEXT NOT NULL,
  command         TEXT NOT NULL,         -- the raw "octos <tool> [args]" string
  status          TEXT NOT NULL,
  received_at     TEXT NOT NULL,
  last_updated_at TEXT NOT NULL,
  UNIQUE(session_id, turn_id)            -- deduplication key
);
```

`command_id` is generated by the API Gateway. The shard never sees it. Dedup key is `(session_id, turn_id)`.

## Ingress Flow

```
onCommand(raw_command):
  if !valid(raw_command):
    return nack("invalid command")

  command_id = newULID()
  parse turn_id, session_id from args

  begin tx
    try insert(command_id, session_id, turn_id, command=raw_command, status="received", ...)
    on duplicate (session_id, turn_id): no-op
  commit tx

  send: octos ack --turn=turn_id --status=stored
```

---

## Failure Scenarios

**Network drop before ACK** — command may already be stored. Shard retries with same `turn_id`. Inbox detects duplicate via UNIQUE constraint. Re-ACKs. No duplicate processing.

**API Gateway crash after receive** — crash before commit: no row, no ACK, shard retries. Crash after commit: row exists, retry gets re-ACK.

**Shard reboot** — command register persists locally. Sender resumes unACKed commands after restart.
