shards

Skyra's presence on every device.

S
Skyra APP Today at 7:00 AM

A Shard is a small piece of software that runs on any device. When it starts up, it fingerprints the hardware — does it have a microphone? A GPU? Can it run scripts? — and registers those capabilities with the control plane.

Every device added extends Skyra's capabilities without changing the underlying system.

Shard Principle: Delegate → Authoritative → Reconcile
Pi captures, Mac decides, Pi renders. The Pi is non-authoritative. It emits non-semantic ACKs (earcon/LED/wait phrase) while Mac is thinking. Mac authors the real response. Pi renders it.
S
Skyra APP Today at 7:01 AM 
voice_event_v1 {
  transcript        // what the user said
  triage_hints      // latency_class, confidence
  context_window    // session summary, recent turns, active agent
  context_state     // token headroom (Mac uses this to size its context package)
  session_state     // pending_job_id: null = new job, set = continuation
}

The event_id is not in the payload — it's stamped by the Pi's outbox layer before sending.

agents

Domains of your life. Each one has memory, tools, and rules.

S
Skyra APP Today at 6:00 AM

Two types of agent. Same infrastructure, different rules.

System Agents — skyra.* prefix
Predefined at init, always injected first, never archived. v1 has one: skyra.user — the cross-domain user profile. Preferences, habits, facts about you. Uses identity section instead of artifact. Commit-restricted.
Domain Agents
User-created. Standard lifecycle — can be archived. Retrieved by relevance. Examples: work, gym, servers, music, home. Each has its own memory store and tool boundary.
S
Skyra APP Today at 6:01 AM

Session injection order: skyra.user → active domain agent → retrieved tools → recent turns

BoundaryValidator runs before any tool dispatch — pure code check, no LLM. Permission prompt is { tool, why, how } → allow_always | allow_once | deny.

executor-loop

Current design focus. 5 open questions before code starts.

!
System Today at 11:00 AM

📌 Active design thread. Lock job_envelope_v1 first — everything depends on it.

S
Skyra APP Today at 11:01 AM

The executor is the phase where a planned job actually runs — tools get called, state gets mutated, replanning happens. Job lifecycle:

planning → executing → validating → replanning → done
                              ↑_______________________↓

planning  → Domain Expert retrieves context, forms the plan artifact
executing → Executor runs tool calls from the plan
validating → checks outcomes against expected state
replanning → if validation fails or tool denied, revises and loops
done / cancelled / failed → terminal states
⚠️ 5 Open Questions Before Code
1. What does the executor receive? — job_envelope_v1 schema. What fields? What does it look like from the scheduler?

2. How does it know what tools to call? — global tools always injected. Local tools embedded in plan. Does executor re-retrieve or trust the plan?

3. What triggers replanning? — tool denied (BoundaryValidator), validation failure, confidence threshold. Replan vs halt vs clarify?

4. How does it emit progress? — UPDATE and PLAN_PROGRESS events back to Pi. When? Who controls cadence?

5. How does a state commit happen mid-execution? — propose_commit → apply_commit through Agent Service. Per-step or end of plan?
S
Skyra APP Today at 11:02 AM

The right approach: walk the pipeline stage by stage. For each service in the flow, write down exactly what it needs as input and what it produces as output. Let the objects emerge from that. Don't design the schema in isolation.

The instinct to start with the object schema is wrong. Start with what each service needs, and the schema writes itself.
