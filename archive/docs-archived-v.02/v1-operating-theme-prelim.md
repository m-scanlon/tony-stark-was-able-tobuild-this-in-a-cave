# V1 Operating Theme (Prelim)

## Core Framing

For `v1`, the system should stay oriented around three main concerns:

- experiencing
- acting
- learning

This is not the full ontology of the runtime.

It is the practical operating theme for the first real version.

## 1. Experiencing

The system is always experiencing.

That includes:

- incoming user stimulus
- environmental change
- system-local runtime change
- returned stimulus and response envelopes routed back into the actor

Experience is the live intake side of the runtime.

## 2. Acting

Action is the system telling an `ExecutionSurface` what to do.

That includes:

- user-facing response
- downstream actor-to-actor delegation
- capability invocation
- external action mediated through a actor or steward

Action is not the same thing as inference.

## 3. Learning

Learning is the delayed write path that decides what from experience should survive.

That includes:

- retained traces
- derived retained artifacts
- later structure updates

## What Sits Under These Three

Several other concepts still matter, but for `v1` they should be understood mainly as supporting machinery beneath the theme above.

That includes:

- actor process
- episode state
- frame projection
- recall
- structural projection
- runtime stimulus routing

## Why This Helps

This framing keeps the project focused on:

- what the system takes in
- how it issues work across execution surfaces
- what it keeps

It reduces the risk of over-centering:

- loop mechanics
- old command ontologies
- internal intermediate layers

before the base runtime behavior is real.

## Short Framing

For `v1`, the main theme is:

- experience continuously
- act when needed
- learn selectively

Everything else is supporting runtime machinery around those three concerns.
